- API .NET 8 principal com 97 endpoints minimal API no Program.cs, JWT, CORS, MySQL, health checks e serviços de integração.

- Docker de produção com API, frontend, Nginx e Watchtower; imagens previstas em GHCR.
- Workflows: GitHub Pages, CI/CD API/frontend, Desktop WPF e pacote Node.
- ERP raiz controller-based com controllers, services, DTOs e models para CRM, financeiro, fiscal, estoque, fornecedores, relatórios e sync.
- ERP GenesisGest separado com Models/DTOs/Services para financeiro, fiscal, estoque e CRM.
- Desktop WPF/PDV com tela principal, PDV de venda, emissão fiscal manual e outbox local.
- Scripts SQL para schema principal, atualização ERP, sincronização de lojas, normalização de enums e schema GenesisGest original.
- Runtime público fixado para https://api.nexumaltivon.com.br e domínio CNAME em nexumaltivon.com.br.

11) ANALISE ESTRUTURAL POR CAMADA
------------------------------------------------------------------------
[Frontend React]
- Projeto CRA/CRACO com React 19, Tailwind, Radix UI, Recharts, Axios e React Router.
- Rotas principais: /, /produtos, /produto/:id, /lojas, /contato, /carrinho, /checkout, /acompanhar-pedido, /login, /area-cliente, /dashboard/*.
- api-runtime.json força API oficial em https://api.nexumaltivon.com.br, com fallback local em desenvolvimento.
- Dashboard.js concentra grande parte do backoffice; arquivo denso, funcional, mas candidato a decomposição modular.

[Back-End API principal]
- Minimal API concentrada em NexumAltivon_Back-End/API/Program.cs, com 10k+ linhas e 97 endpoints.
- Usa EF Core/Pomelo MySQL, JWT Bearer, CORS, DataProtection, Swagger em dev/staging, HTTP clients de integrações e schema bootstrap.
- Ponto forte: fluxo operacional completo de pedido, compras, financeiro, fiscal, logística, CRM e painel corporativo no mesmo processo.
- Ponto crítico: concentração excessiva no Program.cs reduz manutenibilidade e dificulta testes por módulo.

[ERP raiz controller-based]
- Projeto NexumAltivon.ERP.csproj com Program.cs próprio, controllers raiz, Hangfire, Quartz, Serilog/MySQL e DbContext ERP.
- Indica uma segunda API/ERP paralela, controller-based, com endpoints de ERP/CRM/financeiro/fiscal/estoque/sync.
- Precisa decisão arquitetural: manter como serviço ERP separado ou incorporar corretamente na API principal.

[Banco de dados]
- Schema principal nexum_altivon com 23 tabelas, views, procedures e triggers.
- Schema GenesisGest original com 125 estruturas CREATE TABLE e 3 views de ponte Nexum/Genesis.
- Scripts ERP adicionam 22 tabelas ERP, views de resumo, procedures e triggers.
- API principal também cria/garante tabelas operacionais adicionais em runtime, como compras_* e estoque_movimentos.

[Infraestrutura]
- Desenvolvimento: docker-compose sobe MySQL, API, frontend, Redis opcional e Nginx.
- Produção: docker-compose.prod usa imagens GHCR, Nginx e Watchtower; banco é externo por variável API_DEFAULT_CONNECTION.
- GitHub Pages publica frontend .com.br e valida runtime/API; CI/CD também constrói API/frontend e imagens Docker.
- Pendente: TLS explícito, secrets de produção, health de banco real, rollback, backup/restore em produção e monitoramento.

[Desktop/PDV]
- Aplicação WPF net8.0-windows com terminal, cockpit, venda local, outbox JSON e janela de emissão manual NF-e.
- Usa API local padrão http://192.168.1.72:5012 e API pública https://api.nexumaltivon.com.br.

13) RISCOS OBJETIVOS IDENTIFICADOS
------------------------------------------------------------------------
- Concentração de regra de negócio em Program.cs da API principal: risco alto de manutenção e regressão.
- Controllers e serviços existem, mas não são todos compilados pelo csproj da API principal: risco de falsa sensação de endpoint pronto.
- Frontend consome alguns endpoints ausentes no Program.cs principal: risco de erro 404 em produção.
- Segredos reais ou realistas aparecem em arquivos de configuração: risco crítico de segurança; tratar como vazados e rotacionar.
- Produção Docker não inclui banco; depende de banco externo e secrets corretos.
- Nginx de produção expõe apenas HTTP no compose; TLS precisa estar resolvido por proxy externo ou configuração adicional.
- Documentos operacionais referenciados não estão todos presentes no ZIP.
- Integrações externas estão majoritariamente preparadas, mas dependem de credenciais e homologação transacional real.
- API root ERP, API nested e pasta NexumAltivon_ERP indicam três linhas arquiteturais sobrepostas; precisa consolidação formal.
- Não há evidência de suíte automatizada de testes dentro do ZIP analisado.

15) CONCLUSAO OBJETIVA
------------------------------------------------------------------------
O projeto possui uma base enterprise avançada, com front, back, banco, ERP, desktop, CI/CD e infraestrutura já estruturados. O maior ponto de atenção não é ausência de código, mas alinhamento operacional: consolidar qual API é a oficial, garantir que todos os endpoints consumidos estejam publicados no build principal, remover/rotacionar segredos, homologar integrações reais, fechar HTTPS/observabilidade e validar o fluxo ponta a ponta multiempresa: cadastro -> venda -> pagamento -> estoque -> fiscal -> logística -> financeiro -> BI.
